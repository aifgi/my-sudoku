# Progress: difficulty-modes

## Original Goal

Add two difficulty selection modes to the app:
1. **Technique-based** — difficulty determined by which solving techniques are required (current Grader approach: naked singles, hidden singles, naked pairs, etc.)
2. **Given-based** — difficulty determined by the number of given cells in the puzzle

In both modes, the user can choose a difficulty level/grade before starting a game.

Context: The app currently grades all generated puzzles as EASY (per user's observation from the screenshots). The external test (`ExternalPuzzleGraderTest`) shows the engine grades Easy/Medium/Hard/Expert while the reference website uses Easy/Medium/Hard/Expert/Master/Extreme. The current `Difficulty` enum has EASY, MEDIUM, HARD, EXPERT and the `Grader` uses technique-based analysis.

## Learnings

- Given-based generation should stop digging holes at target count (Approach A), not dig-max-then-reject (Approach B) — otherwise Easy (38 givens) would never be generated since max-hole puzzles land at ~22-26
- Engine grades almost everything EASY because the reference site's "Hard" puzzles are actually solvable by the current technique set; the reference site uses a scoring system, not a pure technique-gate
- Critical missing techniques for accurate grading: Naked Triples, Hidden Triples, Swordfish, Y-Wing
- HintEngine gates hints on `difficulty == HARD || EXPERT`; this gating logic breaks for given-based mode and needs decoupling
- No persistence layer exists beyond locale (AppPreferences) — mode preference would need a new AppPreferences key
- Industry consensus given ranges: Easy 36-45, Medium 30-36, Hard 25-30, Expert 22-26, Extreme 17-23 (17 is mathematical minimum, proven 2012)
- ui-redesign spec may change HomeScreen layout — coordinate before finalizing mode toggle placement
- The `Strings` interface + per-locale objects pattern must be followed for all new label keys (English + Russian)

## Learnings (design phase)

- Pointing Pairs must move from HARD to MEDIUM tier — requirements spec defines MEDIUM = Pairs/Pointing Pairs, HARD = Triples; existing GraderTest `hardPuzzle` only needed Pointing Pairs so it will need a new reference puzzle requiring actual Triples
- Given-based ceiling check for SINGLES uses a partial run of the Grader (not full `grade()` call); for PAIRS ceiling, reuse `Grader.grade()` and reject if result is HARD or EXPERT
- `HintResult.NoHintForDifficulty` must never be returned for Given-based mode — existing UI string `hintNoHintForDifficulty` remains valid for Technique mode only
- `DifficultyMode` enum lives in app module (UI concern), not engine module; engine only needs `GivenGrade` and `TechniqueCeiling`
- Mode toggle in HomeScreen reuses the existing `TextButton + alpha()` pattern from the locale toggle — no new Compose components needed
- AppPreferences already uses `Preferences.userRoot().node("sudoku/app")` — mode key `difficulty_mode` slots in alongside existing `locale` key with same error-handling pattern
- `GameViewModel.launchGeneration` is the only site that dispatches to the generator — a single `when(difficulty)` there routes to `generate()` vs `generateByGivenCount()` cleanly
- `HintEngine.findHint` is the only public hint entry point — the `PuzzleDifficulty` signature change propagates only through `GameViewModel.reduce(RequestHint)`
- Swordfish is a 3-row generalisation of X-Wing; implementation requires checking all C(9,3)=84 row triplets per digit — acceptable performance for a puzzle solver
