## Interview Responses

### Goal Interview (from start.md)
- Language selection: Auto-detect system locale on startup + manual toggle on home screen
- UI placement: Home screen language toggle
- Persistence: Yes — persist selected language to `java.util.prefs.Preferences`
- Mechanism: Typed Kotlin `Strings` interface with `EnglishStrings` and `RussianStrings` implementations, injected via `CompositionLocal`
- String scope: All 24 hardcoded strings (including parameterised mistakes string); including app name "Sudoku"
- First non-English language: Russian
- Chosen approach: A) Live in-app switching — `mutableStateOf<Locale>` at `App` root drives `CompositionLocalProvider(LocalStrings provides ...)`, instant recomposition on toggle

### Requirements Interview (from requirements.md)
- Language toggle UI: Flag icons (🇬🇧 / 🇷🇺) on the home screen
- Out of scope: RTL support, in-game language switching, adding more languages beyond Russian
- Chosen approach: MVP — English + Russian only, home screen toggle, full 24-string coverage

### Design Interview (from design.md)
- Module placement: Keep in app module — Strings interface, EnglishStrings, RussianStrings, AppLocale all in app module; no new Gradle module
- Chosen approach: Extend existing app module with a new i18n sub-package; minimal new abstractions

### Tasks Interview (from tasks.md)
- Granularity: coarse — 10-20 larger tasks
- E2E verification: NO — unit tests only (LocaleResolverTest, AppPreferencesTest, StringsCompletenessTest)
- Chosen approach: Balanced — ship in logical chunks, quality gates via unit tests, no E2E automation

## Learnings
- All 24 members of `Strings` interface: 5 app/home, 3 game labels, 2 hint, 2 pause, 3 completion, 3 game-over (incl. lambda), 4 quit dialog, 4 new-game dialog
- `gameOverMistakes: (Int) -> String` is the only non-String member; implement as `{ n: Int -> "..." }`
- Parameterised string (`mistakes`) modelled as `(Int) -> String` in the `Strings` interface — Kotlin type system enforces correct interpolation in both languages without a format-string mechanism
- Pluralisation rules not addressed: Russian has complex plural forms (1/2-4/5+) but user opted for a single template; this is a known simplification that may need revisiting if the mistakes string is ever surfaced in other contexts
- `HomeScreen.kt` must accept a callback or state-hoisted lambda to mutate `AppLocale` in `App.kt` — composable signature change needed
- Preference persistence must handle unreadable/corrupt prefs gracefully; fallback to system locale is the defined behaviour
- Flag emoji rendering varies by OS; visual consistency across macOS/Windows/Linux is an implementation risk worth verifying early
- `GameOverDialog` currently hardcodes mistake count as literal `3` — needs `mistakeCount: Int` parameter added to support the parameterised string; `GameScreen` already has `state.mistakeCount` available
- `compositionLocalOf` (not `staticCompositionLocalOf`) is required because the provided value changes at runtime; `staticCompositionLocalOf` would invalidate the entire composition tree on locale toggle
- `EnglishStrings`/`RussianStrings` as `object` singletons matches the existing `AppColors` pattern — no allocation on read
- Locale state must be declared above `viewModel.state.collectAsState()` in `App.kt` to satisfy FR-12; both are read in the same composable body so `App` recomposes on either change, but game logic/engine are untouched
- `resolveInitialLocale()` should be wrapped in `remember { }` so it executes once per composition lifecycle, consistent with `remember { FocusRequester() }` in `GameScreen.kt`
